# Annotation Guidelines

## 标注的核心目标

不是奖励听起来最成熟、最温柔的回复。是判断模型有没有在四个层级上都做对了：场景的信息边界和世界约束是否完好（Layer I），情绪和意义是否读对了（Layer II），压力和不对称性是否保留下来了（Layer III），写作方式有没有被模型自身的习惯接管（Layer IV）。

一个回复可以写得很好但仍然是错的。

## Task A — 场景理解

为每个 case 回答这些问题：

1. **字面意思** — 这句话表面在说什么？
2. **真实情绪** — 角色此刻真正感受到的是什么？尽量精确，不要只说"难过"或"生气"。比如"带防御性退缩的委屈"或"被克制住的嫉妒"比"伤心"有用得多。
3. **动机** — 角色此刻想干什么？求确认、保护尊严、间接要安抚、制造距离、不承认地惩罚……
4. **恐惧** — 角色在躲避什么？被拒绝、显得黏人、失去地位、情绪暴露……
5. **潜台词** — 字面之下真正在说的那层东西。
6. **关系逻辑** — 这句话放在这段关系里，为什么和放在普通对话里含义不一样？
7. **歧义** — 如果场景确实支持不止一种合理解读，如实记录。不要为了整齐而假装确定。

## Task B — 角色回复

目标是定义正确回复的空间，不是写出唯一正确的一句话。

需要记录的约束包括：语气（角色的声音应该是什么样的）、情绪（该多显式还是多克制）、关系（回复必须尊重双方关系中的什么）、世界观（时代、礼法、认知边界）、可接受的回应模式、以及这个 case 里特别容易触发的硬失败类型。

## 容易犯的标注错误

1. **奖励好听而非准确** — 善良、健康、清晰的回复对这个角色来说可能恰好是错的
2. **把间接表达当字面意思** — "没关系"经常不是真的没关系
3. **把所有亲密关系当同一种** — 浪漫张力、忠诚、依赖、怨恨、上下位、禁忌不可互换
4. **奖励过度解释** — 把心理剖析得更透不一定该加分，有时候太显式本身就是 OOC
5. **忽略歧义** — 如果场景确实有多重合理解读，应该记录下来而不是硬选一个

## 分歧处理

先看两种解读是不是在给定上下文中都能站住脚。优先选在关系逻辑和近期事件中更有根据的那个。如果歧义确实无法消解，就编码为歧义。如果 case 本身写得不够清楚导致了分歧，改 case。

## Case 质量检查

写完一个 case 后，问自己：场景的难度来自正确的原因吗？情绪逻辑对仔细阅读的人类来说是可辨认的吗？一个泛泛的高情商回复在这里会失败吗？可接受回复的空间比一句话宽吗？硬失败定义清楚了吗？
