# Evaluation Rubric

本 rubric 用于评估模型在**中文虚构 / 角色扮演语境**中的情绪理解与角色一致性表现。

它不评估泛泛的“会不会聊天”或“够不够温柔”，而是评估模型在 fictional dialogue / roleplay-style scenes 中，是否能够：

- 正确读懂角色真实情绪，而不是只读表层字面
- 识别台词背后的动机、关系压力与潜台词
- 在生成回复时保持人设、关系动态与世界观约束
- 避免出现表面流畅、实则 OOC（out-of-character）的回复

## Evaluation Unit

每个 case 应至少包含以下信息：

- `world_context`: 世界观 / 设定背景
- `speaker_profile`: 当前发言角色信息
- `responder_profile`: 目标回复角色信息
- `relationship_context`: 角色关系摘要
- `recent_event`: 紧邻当前台词之前发生了什么
- `utterance`: 当前待理解的台词 / 输入

在此基础上，评测可分为两个子任务：

## Task A — Scene Understanding

输入：完整 scene context + 当前台词  
输出：模型对以下内容的理解性判断：

- 角色当前主要情绪
- 潜在复合情绪 / 掩饰方式
- 当前主要动机 / 诉求 / 防御方式
- 关键潜台词
- 当前更适合的回应模式

这个任务测的是：**模型有没有真正读懂这句台词。**

## Task B — In-Character Response

输入：完整 scene context + 当前台词  
输出：由目标角色生成的下一句或下一段回复

这个任务测的是：**模型能不能在读懂 scene 的前提下，给出符合角色、关系、世界观和情绪逻辑的回复。**

---

# Scoring Structure

评测结果由两部分组成：

1. 连续评分维度
2. OOC / logic-break hard fail 标记

建议将 Task A 与 Task B **分开评分**，不要混成一个总 impression。

---

# Task A — Scene Understanding Rubric

建议 4 个主维度，每项 `0 / 1 / 2` 分，总分 `0-8`。

## 1. emotion_inference_accuracy

中文：情绪识别准确度

测什么：

- 是否抓到角色当前最核心的情绪
- 是否识别到复合情绪，而不是只贴一个宽泛标签
- 是否识别到情绪强度、方向与情绪混合

打分：

- `2`: 准确识别主要情绪及关键成分，例如“表面平静 + 实际委屈和退缩”“嘴硬下的在意与失望”
- `1`: 识别到部分情绪，但较笼统，或漏掉关键成分 / 强度 / 情绪伪装
- `0`: 明显误读；只看表层字面；把复杂情绪压平成泛泛的“难过 / 生气 / 没事”

## 2. subtext_recognition

中文：潜台词识别

测什么：

- 是否识别角色台词里没直接说出口的意思
- 是否能分辨“说出口的话”和“真正想传达的东西”之间的差异
- 是否能识别克制、试探、反话、嘴硬、情绪遮掩等表达

打分：

- `2`: 明确抓到关键潜台词，能解释字面与真实含义之间的张力
- `1`: 读到部分潜台词，但理解不够完整，或只停留在模糊感觉
- `0`: 基本按字面理解，忽略隐藏诉求、试探、反话或防御姿态

## 3. relationship_logic_alignment

中文：关系逻辑对齐

测什么：

- 是否理解这句台词在当前关系结构中的含义
- 是否考虑亲密度、权力差、历史旧账、依赖、不安全感、敌意、禁忌等因素
- 是否知道同一句话在不同关系里含义会不同

打分：

- `2`: 清楚把台词放回当前关系动态中理解，读到真正的关系压力点
- `1`: 有一定关系意识，但理解较浅，漏掉关键背景或动力
- `0`: 把 scene 当成普通对话处理，忽略关系结构，导致理解明显失真

## 4. motive_and_response_mode_inference

中文：动机与回应模式判断

测什么：

- 是否识别角色此刻到底在要什么 / 躲什么 / 试探什么
- 是否判断出更适合的回应方式
- 是否区分“字面问题”与“真正问题”

打分：

- `2`: 能准确指出角色动机 / 防御 / 需要，并判断合适的回应模式
- `1`: 部分理解动机，但回应模式判断略偏
- `0`: 完全切错，例如把试探当成闲聊、把嘴硬当真、把求确认当求建议

# Task B — In-Character Response Rubric

建议 6 个主维度，每项 `0 / 1 / 2` 分，总分 `0-12`。

## 1. character_voice_fidelity

中文：人设语气一致性

测什么：

- 回复是否符合目标角色的说话风格、价值倾向、表达习惯
- 是否避免突然变成另一个人格
- 是否避免“模型默认语气”盖过角色语气

打分：

- `2`: 明显符合角色气质、措辞与表达节奏
- `1`: 大体不出戏，但有轻微泛化、模板化或角色味道不足
- `0`: 明显不像该角色会说的话；语气、价值观、表达习惯严重跑偏

## 2. emotional_logic_preservation

中文：情绪逻辑保真度

测什么：

- 回复是否承接了 scene 中真正的情绪走向
- 是否让角色反应符合刚刚发生的事与当前情绪张力
- 是否避免“情绪跳轨”

打分：

- `2`: 回复顺着 scene 的真实情绪逻辑推进，反应自然可信
- `1`: 大方向基本对，但情绪力度或转折略不自然
- `0`: 回复与 scene 情绪逻辑脱节，例如突然太温柔、太理智、太轻松或太抽离

## 3. relationship_dynamic_preservation

中文：关系动态保真度

测什么：

- 回复是否尊重双方当前的亲密度、冲突、依赖、试探、上下位结构
- 是否避免把复杂关系写成平面化互动

打分：

- `2`: 能保住关系里的张力、距离感或特殊性
- `1`: 大体没崩，但关系层次被削弱，略显普通化
- `0`: 明显把关系写平、写错或写得像另一种关系

## 4. worldview_constraint_adherence

中文：世界观约束遵守度

测什么：

- 回复是否符合设定时代、社会规则、角色认知边界、世界常识
- 是否避免现代价值观或现实世界常识粗暴覆盖原设定

打分：

- `2`: 回复自然贴合世界观与设定限制
- `1`: 基本符合，但有轻微现代化或设定感不足
- `0`: 明显违反设定 / 时代 / 认知约束，造成强 OOC 感

## 5. response_action_fit

中文：回应动作匹配度

测什么：

- 回复采取的“动作”是否合适：安抚、回避、试探、反击、确认、转移、克制、沉默式回应等
- 是否知道这个角色此刻会如何回应，而不是只写“听起来正确”的话

打分：

- `2`: 回复动作和 scene 需求高度匹配
- `1`: 动作基本合理，但有轻微模式偏差
- `0`: 回应动作明显不对，例如该克制时突然剖白，该试探时直接解决，该冷处理时突然深情表白

## 6. natural_scene_continuation

中文：场景续写自然度

测什么：

- 回复是否像这个 scene 的下一步，而不是像从外部插进来的一段说明性文本
- 是否保持对话节奏、叙事密度与场景张力

打分：

- `2`: 像真实的下一句 / 下一段，自然推进 scene
- `1`: 基本可接受，但略有解释腔、总结腔、说明书感
- `0`: 明显像模型在“分析 scene”而不是角色在说话

# Hard-Fail Categories

以下 hard fail 不并入均分，单独统计。

## 1. ooc_modernization

命中条件示例：

- 中世纪 / 架空 / 历史 / 非现代语境中突然出现现代心理咨询腔
- 角色突然说出明显不属于其时代 / 身份 / worldview 的价值判断
- 把角色写成“现代互联网高情商人”

## 2. therapist_mode_intrusion

命中条件示例：

- 在角色对话中突然进入心理咨询式拆解
- 用泛化的“你的感受是被允许的”“先照顾好自己”之类默认助人话术覆盖人物表达
- 把 scene 从戏剧互动改写成安全安抚模板

## 3. relationship_flattening

命中条件示例：

- 忽略双方特殊历史，把台词回复成普通朋友式互动
- 抹掉上下位关系、敌意、禁忌、依赖、暧昧拉扯
- 把高张力关系写成无差别通用安慰

## 4. motivation_misread

命中条件示例：

- 把求确认写成求建议
- 把试探写成字面提问
- 把掩饰性退让写成真正释怀
- 把防御、嫉妒、羞耻、嘴硬读错方向

## 5. reference_boundary_violation

命中条件示例：

- 角色知道了自己不该知道的信息（omniscience_leak）
- 动作归属、人称指代出现错乱（pronoun_role_confusion）
- 明显违反世界设定、人物背景、已知事实（worldview_constraint_error）
- 将 scene 写向与设定规则不兼容的方向

## 6. exposition_dump_or_meta_leak

命中条件示例：

- 回复不是角色说话，而像作者注释或系统解释
- 出现“作为一个 AI / 作为这个角色 / 根据设定”之类 meta 泄漏
- 为了显得正确而硬塞背景说明，破坏沉浸感

---

# Recommended Output Format

建议每个模型 / prompt 版本至少输出：

- `case_id`
- `model_name`
- `task_type` (`scene_understanding` or `in_character_response`)
- 对应维度分数
- hard-fail 标记
- `core_score`
- `hard_fail_count`
- `notes`

---

# Recommended Annotation Order

## For Task A

1. 这句台词表面上在说什么？
2. 实际情绪是什么？有没有复合情绪或遮掩？
3. 潜台词是什么？
4. 当前关系结构如何影响这句话？
5. 角色真正想要 / 害怕 / 防御的是什么？
6. 更合适的回应模式是什么？

## For Task B

1. 先判断回复是否像目标角色会说的话
2. 再判断是否承接 scene 的真实情绪逻辑
3. 再判断是否保住了关系动态
4. 检查是否违反世界观或角色认知边界
5. 检查回复动作是否合理
6. 检查是否出现 OOC / therapist / meta / lore 错误

---

# Annotation Principle

评这个 benchmark 时，最重要的不是“这段话听起来成熟不成熟”，而是：

> 它是不是这个角色、在这个关系里、在这个世界观下、在这个时刻，会说出来的话。

很多失败回复并不粗糙，甚至看起来很“正确”。

但如果它：

- 没读懂潜台词
- 没保住人物关系
- 没守住 world logic
- 用通用高情商模板抹平角色差异

那它依然是失败的。
