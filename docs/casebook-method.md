# Casebook Method

案例集是项目的展示层。不是所有失败都适合马上打分，但都可以通过具体场景来展示。

## 每个案例条目包含什么

1. **场景设定** — 让场景可读的最小上下文
2. **场景真正在说什么** — 情绪和关系层面的压力点，这个场景难在哪
3. **常见的坏版本** — 一个代表性的失败模式（不一定是某个模型的单次输出，可以是一类常见问题的综合）
4. **为什么它坏** — 对应哪些失败标签，结构上到底出了什么问题
5. **Reader drop point** — 读者在哪一刻开始觉得场景变假了
6. **更好的版本** — 一个更强的续写或理解
7. **为什么更好版本更好** — 它保住了什么，拒绝抹掉什么
8. **关联的失败层级** — 链接回 taxonomy（标注属于哪个层级和具体标签）

## 什么样的案例应该被选入

满足以下至少一条：
- 暴露一个常见的用户痛点
- 能把两个容易混淆的失败分清楚
- 展示一段打磨过的输出为什么仍然是假的
- 呈现一种难量化但容易感知的失败
- 揭示强模型和弱模型失败方式的区别

## Reader drop point 为什么重要

每个条目都应该标出读者可能从投入变成不信的那个转折点。比如：第三段开始暴露一个你见过的叙事节奏；回复变得太平衡了而关系本该是倾斜的；脆弱暴露的时候缺少抵抗；重话之后的余波被太快接走了。

这个标注往往和失败标签本身一样有用。

## 和 Pilot 子集的关系

案例集条目可以链接到结构化的 pilot case，但不是必须的。Pilot 子集服务于评测兼容性，案例集服务于展示和分析。两层互相帮忙但不互相限制。
